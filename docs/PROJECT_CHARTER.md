# Project Charter

## 项目名称

LYGD AI Tunnel Platform

## 项目使命

建立一个受业务规则约束、可审计、可逐步演进的隧道二衬台车方案与报价辅助平台，降低非标项目参数遗漏、经验数据混用和报价过程不可追溯的风险。

## 项目目标

- 统一项目、隧道、台车、成本和智能配置输入。
- 明确经验重量、工程师确认重量和正式设计重量之间的边界。
- 形成可测试、可维护的重量与报价计算模块。
- 建立产品文档、变更记录和 Sprint 审核机制。
- 为后续数据版本、报价历史和审批能力保留架构空间。

## 范围

### 范围内

- 隧道二衬台车项目参数采集。
- 经验重量参考及人工确认。
- 成本汇总、利润计算和建议报价辅助。
- 国内与海外报价场景的基础信息记录。
- 产品治理、业务规则、架构和变更文档。

### 范围外

- 结构设计、强度验算和施工图出图。
- 自动生成正式合同或未经审批的正式报价。
- 在没有权威数据来源时自动推断价格。
- 取代工程师、财务、商务或管理人员的审批责任。

## 关键角色

- 产品负责人：确认产品目标、Sprint 范围和优先级。
- 业务负责人：批准重量、价格、利润及报价规则。
- 工程师：确认非标参数与最终机械主体重量。
- 开发人员/AI 代理：在批准范围内实现并测试，不得改写业务规则。
- 审核人员：核对变更、测试、已知问题和文档同步情况。

## 治理原则

1. 业务规则优先于技术便利。
2. 数据来源必须可说明，经验数据不得冒充设计数据。
3. 重大变更必须可追溯并写入 CHANGELOG。
4. 每个 Sprint 独立验收，未经审核不得自动进入下一 Sprint。
5. 采用最小变更原则，保持既有业务页面和已验证行为。

## 成功标准

- 已批准业务规则在代码、文档和测试中保持一致。
- 核心计算可以独立测试。
- 每个版本能追溯修改文件、测试结果和已知问题。
- 用户能够明确区分参考结果、人工确认结果和正式设计结论。
