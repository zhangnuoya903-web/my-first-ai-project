# Sprint Plan

## Sprint 治理规则

每个 Sprint 必须具备明确目标、范围、禁止事项和验收条件。完成时必须输出修改文件、测试结果、已知问题、功能影响和 Git Commit。未经人工审核，不得自动进入下一 Sprint。

## Sprint 0 — Project Governance

目标：建立产品治理体系和 AI 开发约束。

交付：项目章程、产品愿景、路线图、业务规则、系统架构、Sprint 计划、变更日志、README 和 AGENTS。

状态：已完成，等待审核。

## Sprint 1 — V2.0 Recovery

目标：恢复 V2.0 页面计算与重置功能。

交付：DOM ID 对齐、计算按钮、重置按钮、事件绑定和回归测试。

状态：已完成。

## Sprint 2 — JavaScript Modularization

目标：在不改变 UI 和公式的前提下完成 ES Modules 拆分。

交付：应用入口、重量引擎、报价引擎、UI 控制器和存储预留接口。

状态：已完成。

## Sprint 3 — Data Model + Business Rules Foundation

目标：建立项目、报价、业务规则、重量参考和智能配置的机器可读数据基础。

交付：5 个 data JSON 文件、规则编号同步、数据目录文档和一致性测试。

状态：已完成，等待审核。

## Sprint 4 — 数据版本治理

候选方向：数据来源、版本、生效日期和维护责任。

状态：未开始。

## Sprint 5 — 持久化基础

候选方向：项目、客户和报价记录的数据模型。

状态：未开始。

## Sprint 6 — 报价生命周期

候选方向：报价保存、查询、复制和版本快照。

状态：未开始。

## Sprint 7 — 审批与审计

候选方向：人工修正、审批状态和审计日志。

状态：未开始。

## Sprint 8 — 输出与交付

候选方向：受控打印、报价明细和文档导出。

状态：未开始。

## Sprint 9 — 质量体系

候选方向：单元、集成、端到端测试及持续集成。

状态：未开始。

## Sprint 10 — 上线准备

候选方向：权限、备份、监控、部署与运维文档。

状态：未开始。
